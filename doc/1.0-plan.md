# VFXLand5 1.0 Planning

Goal: Finalize forge and engineer APIs. Ship supershow as a coherent 2D pixel art game engine. Use lib/ approach for development with snapshots for dormant projects.

## Repository Identity

- "vfxland5" = era/workspace name, not a package
- "supershow" = the actual product (2D pixel art engine)
- forge/engineer = implementation details of supershow, bundled together
- Consider renaming repo to "supershow" to clarify identity

Don't architect for hypothetical future engines (e.g., 3D):
- forge has fixed-point math baked in - that's fine, it's supershow-flavored
- If a 3D engine ever happens, fork what's needed into a separate repo
- Speculative generalization isn't worth the complexity for a solo project

## Dependency Principle

**Never nest projects with the same submodules** (diamond dependency problem).

Bad:
```
game/
    supershow/        <- copy 1
    addon/
        supershow/    <- copy 2 (conflict, search confusion, wrong-file edits)
```

Solutions:

| Pattern             | How it works                                    |
|---------------------|-------------------------------------------------|
| Bundle together     | tightly-coupled components live in same repo    |
| Source-only layer   | addon has no submodule, assumes engine loaded   |
| Shared lib          | all projects reference same lib/ copy           |
| Top-level owns deps | only game has submodule, addons plug in         |

Any dependency should exist in **exactly one place** in the tree.

## Architecture

```
supershow/            The distributable engine (repo)
    forge/            Foundation (language extensions, Allegro bindings, fixed-point math)
    engineer/         Runtime shell (window, loop, REPL, hot-reload)
    src/              2D game engine (actors, sprites, tilemaps, audio)
    workzone/        GUI toolkit and IDE/REPL ecosystem
```

Principles:
- forge/engineer/workzone = bundled components, versioned together
- supershow = curated collection of modules with defined interfaces
- Modules isolate namespaces; swappable components (e.g., audio) can coexist
- Breaking changes require 2.0

## Workspace Structure

```
vfxland5-work/                      Era/workspace folder (not a repo)
    lib/
        supershow/                  Stable release - all projects reference this
            forge/
            engineer/
            src/
            workzone/
    supershow-dev/
        supershow/                  Active development
        tests/
        experiments/
    games/
        darkblue/                   Uses lib/supershow via %LIB% macro
        kvn/
        ...
    addons/                         Source-only, no submodules
        asu/
```

**Workflow:**
- Daily development: games reference lib/supershow, clean search, one copy
- Ship/archive a project: snapshot lib/supershow into project folder, remove from main workspace
- Return to dormant project:
  - Quick fix: open standalone, don't add to workspace
  - Real update: migrate to latest lib/supershow, bring back into workspace

**WorkZone:** Moves into supershow repo. Tightly coupled, versioned together, both potentially needed at runtime.

**lib/ vs addons/:**

| Folder   | Contains                                      | Can depend on               |
|----------|-----------------------------------------------|-----------------------------| 
| lib/     | Self-contained packages, versioned independently | Other lib/ packages (including forge) |
| addons/  | Supershow extensions, part of ecosystem       | lib/ (any supershow layer)  |

Examples:
- `lib/supershow/` - the game engine
- `lib/fixedvec/` - standalone fixed-point vector math (uses forge, but independent package)
- `addons/player2/` - AI NPC integration (extends supershow)
- `addons/spunk/` - room-grid world system (extends supershow)

The test: **Could this be versioned independently?**
- Yes → lib/
- No, it's an extension of the supershow ecosystem → addons/

Addons can depend on any supershow layer (forge, engineer, or full engine). A forge-only addon is still an addon if it's extending the ecosystem rather than being a standalone reusable package.

### forge/lib/strout.vfx
Status: REWORK API
- Captures console output (emit, type, cr) into string via `s{` ... `s}`
- Used by: engineer/window.vfx, supershow/stage.vfx, spunk editors, kvn/player2
- Largely superseded by format.vfx for simple cases
- Problems with current implementation:
  - Wastes memory (4 x 32KB rotating buffers)
  - Limited re-entrance
  - Hard limit on output size
- New API (strout2.vfx):
  ```
  strout ( xt - )
  str@   ( - a len )
  ```
  - Allocates 256-byte heap buffer, grows automatically as needed
  - Redirects console output to buffer
  - Executes xt; use `str@` inside to get captured output as addr/len
  - Automatically frees buffer after xt returns
- Example:
  ```forth
  [: room# ? str@ print ;] strout
  ```
  - Easy to pass context to xt
  - Consumption optional
- Needs migration of existing callers

## Module system improvements

Goal: Enable namespace isolation for:
1. Vertical isolation - top-level programs (games, editors, tools) don't leak into each other, even when integrated into the same executable
2. Horizontal isolation - swappable modules within a program coexist without collision

Implementation:
- Store entire search order at module creation time
- Each module sees only what it declares as dependencies

## fstools detailed analysis

### Inconsistent patterns
- `read` / `_?read` / `?read` / `load-file` - four ways to read files, each with different error handling
- `write` vs `write{`/`data,`/`write}` - two paradigms for writing
- Error handling is a mix of `abort"`, `throw`, `.abort`, and silent catch

### Noise
- `write`, `read`, `load-file` all print to console unconditionally

### Naming
- `-file` and `ext` are cryptic
- `file,` vs `load-file` - inconsistent conventions

### Global state
- `#1` value used by the write{ } words - not reentrant

### Hardcoded assumptions
- `find-dat-path` knows about `dat` and `..\` - project-structure-specific, may belong in supershow

### Candidate words to keep (pending review)

| Word          | Purpose                         |
|---------------|---------------------------------|
| load-file     | Read file to allocated memory   |
| write         | Write buffer to file (rename?)  |
| file-exists?  | Check file existence            |
| dir-exists?   | Check directory existence       |
| fsize         | Get file size                   |
| ensure-dir    | Create directory                |

### Candidate words to rethink

| Word            | Issue                                           |
|-----------------|-------------------------------------------------|
| find-dat-path   | Move to supershow? It's game-project-specific   |
| write{ } family | Decide: keep streaming API or drop it?          |
| read            | Overlaps with load-file                         |
| -file / ext     | Rename to be clearer, or drop                   |
| ?read / _?read  | Clarify purpose or consolidate                  |
